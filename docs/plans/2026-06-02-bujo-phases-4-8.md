# BuJo Phases 4-8 Implementation Plan

> **For Hermes:** Use subagent-driven-development skill to implement this plan task-by-task.

**Goal:** Finish BuJo from Phase 4 validation through Phase 8 command palette with strict TDD, no hidden rewrites, and no new LLM feature work before the data model is fixed.

**Architecture:** Keep markdown files as the durable source of truth. Add pure modules for parsing, analytics, habits, and AI provider logic so Electron IPC is thin and UI components are mostly render/orchestration. Phase 5 changes the in-memory entry model but preserves existing markdown symbols for backward compatibility.

**Tech Stack:** Electron, Vite, React 19, TypeScript, Vitest, local markdown vault, direct MiniMax/OpenRouter-compatible chat completions.

---

## Current State Baseline

**Known state as of this plan:**

- Phase 4 is mostly implemented but under-tested and the 14-day habit matrix does not load persisted completions.
- Phase 5 data model fix is not started.
- Phase 6 LLM features are not started, except provider plumbing now supports MiniMax direct and OpenRouter fallback.
- Phase 7 is partially done: `electron/analytics.ts`, `electron/parser.ts`, and `electron/aiProvider.ts` exist, but renderer service expansion is not done.
- Phase 8 command palette is not started.

**Baseline commands that must pass after every phase:**

```bash
npm test
npm run lint
npm run build
```

Expected final baseline: all tests pass, TypeScript passes, production build succeeds.

---

## Execution Rules

1. Use TDD for all behavior changes.
2. Write the failing test first, run it, confirm it fails for the expected reason.
3. Implement the smallest change that passes.
4. Run the targeted test.
5. Run `npm test && npm run lint && npm run build` at the end of each phase.
6. Do not start Phase 6 LLM features until Phase 5 is complete.
7. Do not rewrite existing markdown journal files. Parser mapping handles old symbols.
8. Keep OpenRouter as fallback, but prefer direct MiniMax when `MINIMAX_API_KEY` is available.

# Phase 5: Data Model Fix

**Phase goal:** Separate entry kind from task status while preserving old markdown file format.

## Target Model

In `src/types.ts` and renderer-facing types:

```ts
export type EntryKind = 'task' | 'note' | 'event'
export type EntryStatus = 'active' | 'done' | 'killed' | 'migrated'

export interface EntryMeta {
  priority?: boolean
  scheduledFor?: string
  migratedTo?: string
}

export interface Entry {
  id: string
  kind: EntryKind
  status: EntryStatus
  content: string
  timestamp: number
  source_date?: string
  display?: string
  meta?: EntryMeta
}
```

## Mapping Rules

- `task` -> `kind: task`, `status: active`
- `done` -> `kind: task`, `status: done`
- `migrated` -> `kind: task`, `status: migrated`
- `killed` -> `kind: task`, `status: killed`
- `note` -> `kind: note`, `status: active`
- `event` -> `kind: event`, `status: active`
- `scheduled` -> `kind: task`, `status: active`, `meta.scheduledFor`
- `priority` -> `kind: task`, `status: active`, `meta.priority: true`
